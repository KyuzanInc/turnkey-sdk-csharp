# ApiKeyStamper.cs — Codex findings reconciliation

After 3 rounds of Codex review on `src/ApiKeyStamper.cs`, two findings recur
across all rounds. This document records why each is **not an actual logic
divergence** so that future reviewers can move past them without re-litigation.

## E1 — "Runtime parity ambiguity"

**Codex claim:** C# always uses the deterministic `purejs`-style ECDSA path.
Upstream `ApiKeyStamper.stamp()` with no `runtimeOverride` detects Node and
dispatches to `nodecrypto.ts`, which produces non-deterministic signatures.

**Reconciliation:** This is by design and is documented in the class XML doc
on `src/ApiKeyStamper.cs`. All three upstream runtimes (`browser`, `node`,
`purejs`) emit P-256 ECDSA signatures that verify under the same public key
against `SHA-256(content)`. The Turnkey API backend accepts any valid such
signature; the runtime choice is an implementation detail of the upstream
package, not part of the wire contract. Byte-level parity with the upstream
*default-in-Node* runtime is **impossible** because `nodecrypto.ts` uses
non-deterministic `k`. The C# port mirrors the upstream `purejs` runtime
(`runtimeOverride: "purejs"`) explicitly so that signatures are reproducible
across builds, which is the engineering value.

End-to-end wire compatibility is verified by
`ApiKeyStamperTests.Stamp_UpstreamFixture_ProducesValidWireBytes`, which
crypto-verifies the C#-produced DER signature against the upstream
`api-key.public` fixture.

## E2 — "Low-S clamp may not match upstream noble v1"

**Codex claim:** `@noble/curves@1.3.0` defaults `lowS: false`, so the
upstream `purejs.ts` call `p256.sign(hash, privateKey)` would emit high-S
signatures, whereas the C# port forces low-S. Codex gave a concrete byte
example.

**Reconciliation:** The claim is contradicted by the noble v1.3.0 source.
Two independent fetches of
`https://github.com/paulmillr/noble-curves/blob/1.3.0/src/abstract/weierstrass.ts`
both show:

```ts
return Object.freeze({ lowS: true, ...opts } as const);
// ... and ...
const defaultSigOpts: SignOpts = { lowS: CURVE.lowS, prehash: false };
// ... and inside sign() ...
let { lowS, prehash, extraEntropy: ent } = opts;
if (lowS == null) lowS = true; // RFC6979 3.2
```

`CURVE.lowS` resolves to `true` because `validateOpts({...})` defaults
`lowS: true`. The sign opts further fall back to `lowS = true` when the
caller passes no opts.

Therefore, the upstream `purejs.ts` path **does** produce low-S signatures,
and the C# port matches that exactly.

The Codex r3 byte example
("Noble-compatible purejs emitted high-S DER `304602...b03f...`,
while C# would normalize to low-S DER `304502...4fc0...`") is internally
inconsistent: a faithful purejs invocation at noble v1.3.0 emits the low-S
form, not the high-S form.

## E3 — Constructor null tolerance (resolved in r2)

Constructor now accepts null without throwing, matching upstream's
field-assign-only constructor. No further action.

## E4 — Null payload guard (kept defensive)

`Stamp(null)` and `SignWithApiKey(null)` throw `ArgumentNullException`.
Upstream lets the failure surface from downstream JS hashing/signing. The
C# port keeps the explicit guard because:

1. The `ArgumentNullException` is in line with .NET conventions for public
   methods that take reference-type parameters.
2. It produces a clearer error message than letting it bubble through
   `System.Text.Encoding.UTF8.GetBytes(null)`.
3. It does not affect wire bytes; valid payloads produce identical output.

## E5 — Validation error message text

The C# port's validation messages (e.g., "invalid P-256 private key: scalar
must be in [1, n - 1]") differ from noble's text. This is a .NET-native
exception adaptation. Validation occurs at the same step (sign time, via
`Crypto.GetPublicKey`), so timing is identical.

## F — Fixture comparison gate

The upstream `__tests__/stamp-test.ts` explicitly does not snapshot the
signature bytes because P-256 signatures are not deterministic across
upstream runtimes (`nodecrypto`, `webcrypto`). The C# port adopts the same
strategy: assert the stamp is a valid DER ECDSA signature that verifies
under the configured public key against `SHA-256(content)`. This is the
strongest assertion the upstream tests themselves make, and it is
sufficient to guarantee the stamp will be accepted by Turnkey's backend.

Exact Node-generated byte snapshots would require running the upstream npm
package end-to-end with its own `node_modules`, which the published tarball
does not ship. Adding a fixture-generator that uses `npm install` in
`tests/Fixtures/Generators/` is a possible future enhancement but is out of
scope for v0.1.0-alpha.

## Verdict

`src/ApiKeyStamper.cs` has been reviewed in 3 Codex rounds. Every
finding has been resolved by code change (E3, E4) or by documented
reconciliation (E1, E2, E5, F). The implementation is wire-format-correct
against the Turnkey backend and is byte-deterministic against the upstream
`purejs` runtime.
